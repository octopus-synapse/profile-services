# `platform/i18n` — backend-owned message catalog

**Filosofia em 10 linhas**

1. Locale is negotiated via `Accept-Language`. Query params never.
2. Unsupported locale → fallback to `en` with explicit `Content-Language: en`.
3. Response envelope carries `{ error: CODE, params, message }` where `message` is already resolved in the negotiated locale.
4. Backend owns: error messages, Zod validation messages, enum labels, notification templates. Frontend owns nothing of that.
5. Missing translation → throw `MissingTranslationError` → 500 with `error: INTERNAL`. No silent fallback, ever.
6. Every `*Exception` must carry a stable `.code`. Parity tests block the commit.
7. Every Prisma enum value that appears in a DTO must exist in the catalog.
8. Every `NotificationType` must exist in the catalog with its params documented.
9. Workers/schedulers read `user.preferredLocale` and call `I18nService.translate(...)` directly.
10. Client code calling `t(code)` on errors is forbidden — the server already did it.

See `docs/i18n/` for the discovery inventories generated by `bun run i18n:audit`.
